Bench 在做什么

Interaction2Code 关注的是从交互原型截图生成可交互网页，而不是只复刻静态页面。

127
unique webpages
覆盖不同网页主题和前端框架
374
distinct interactions
论文/README 标注规模；本地 hub 有 375 条 case
15
webpage types
商品、博客、文档、工具、系统页等
31
interaction categories
点击、悬停、输入、选择、新页面、新组件等
ASE
论文发表
40th IEEE/ACM ASE 2025
输入
同一网页交互前后的截图：例如 0.png 表示原始状态，1.png 表示执行第 1 个交互后的状态；Mark prompt 还会给交互元素和变化区域画红框。
输出
一个可运行网页 HTML，初始状态要像 before 图；用户触发指定元素后，页面状态要像 after 图。官方 prompt 要求触发元素使用 id="interact1"。
核心难点
模型既要复刻静态视觉，又要理解交互意图、绑定正确页面元素、写出脚本状态变化，并让截图链路能捕获交互后的视觉状态。

官方链路

从输入截图到最终分数，中间至少经过生成、渲染、触发、截图和指标计算五件事。

1

读取数据

每个网页目录包含多张交互原型图和 action.json。配置里记录 src/dst、交互动作、tag type 和 visual type。

2

模型生成

MLLM 接收 before/after 或 mark 图，生成完整 HTML/CSS/JS。Mark prompt 强制把触发元素标为 interact1。

3

浏览器执行

官方代码用 Selenium/Firefox 打开 HTML，保存 0_source.png，再滚动到交互元素并点击，保存 1_x_y_click.png。

4

指标计算

先用 CLIP 从点击截图中找最接近目标 after 图的一张；如果 evaluator 判断交互有效，再计算 Text、SSIM、Position 和 interaction 区域指标。

评测指标

论文主表分为 Full Page 和 Interaction Part；README 显式展示的是 Interaction Part leaderboard。

指标 作用域 含义 需要注意
CLIP Full / Interaction 整体视觉语义相似度 proxy。 对组件主题和视觉大意敏感，但不保证像素级布局一致。
SSIM Full / Interaction 低层结构、边缘、布局相似度 proxy。 会被截图尺寸、滚动、字体、缩放影响，不是纯 layout。
Text Full / Interaction OCR 后的文本相似度。 文字没抄准、OCR 失败、截图没截到区域都会拉低。
Position Interaction 交互元素位置与红框标注位置的相似度。 官方代码使用点击图文件名里的元素坐标和 mark 图红框位置计算。
IR Interaction Implement Rate，交互是否被 evaluator 判定为实现。 缺网页文件、无点击后截图、交互前后图相同等会导致 IR 失败或计 0。

可优化问题总结

这些不是单纯的 case 现象，而是可以组织成方法动机的失败类型。优先级越靠前，越适合作为论文主线；后面的点更适合作为补强模块或实验分析工具。

全局布局和比例不稳

主线 1

模型能识别组件，但经常把左右结构生成成上下结构，主内容、侧栏、卡片和图片比例也不稳定。

表现
页面大结构错、组件尺寸错、间距错，Full Page 的 CLIP/SSIM 都容易被拉低。
方向
先显式抽取页面骨架、区域比例和组件层级，再约束代码生成。

交互状态转移理解不足

主线 2

模型不一定理解 before 图到 after 图之间发生了什么，经常只复刻某一张静态图。

表现
点击后本应换页面、展开菜单、选择日期范围或更新筛选结果，但生成页面没有形成正确状态变化。
方向
把任务显式拆成初始状态、用户动作、目标状态、变化区域和转移类型。

缺少渲染反馈和自检

主线 3

当前生成更像单轮写代码，写完后没有打开页面检查，也没有根据截图差异修正。

表现
HTML 截断、空页面、交互没生效、截图为空，往往要到评测阶段才暴露。
方向
生成后自动渲染、截图、检查空白和变化区域，再进行一到多轮修复。

触发器绑定和交互代码易错

主线 4

模型经常不是完全没写交互，而是事件绑定、状态类名、选择器和目标节点没有对上。

表现
例如样式要求菜单节点加 active，但脚本把 active 加到父节点，点击后视觉状态没有变化。
方向
对生成代码做事件绑定检查、目标元素检查和触发后状态差异检查。

视觉素材还原能力弱

补强

图片、图标、头像、商品图、logo 和 banner 经常被灰块、蓝块或空白占位替代。

表现
页面主题和视觉重心丢失，真实图片比例没有被复现，布局也会被带偏。
方向
先定位关键视觉素材，区分可复用素材、可近似素材和必须生成的占位。

字体、字号和文本密度不准

补强

很多页面结构大体正确，但一眼看起来不像，原因往往在文字系统。

表现
标题过大或过小、行高不对、字重不对、换行方式不对，导航和卡片文字密度偏差明显。
方向
从截图估计字号层级、行高、字重和文本块密度，作为代码生成约束。

组件层级和浮层位置不稳

补强

弹窗、下拉菜单、tooltip、hover 卡片和侧边栏这些浮层组件经常位置或遮挡关系错误。

表现
应该覆盖的没有覆盖，应该贴着触发按钮的浮层跑偏，或者 z-index 不正确。
方向
显式建模浮层锚点、相对位置、覆盖范围和层级关系。

页面任务语义理解不够

补强

有些错误不是某个按钮错了，而是模型没看懂这个页面到底在完成什么业务动作。

表现
后台管理页写成展示页，筛选流程只做成静态表单，多状态页面只还原了一个状态。
方向
先判断页面类型和交互意图，再决定需要实现的组件状态和数据变化。

生成完整性和源码异常

质检

部分任务不是审美失败，而是代码文件本身不完整或输出内容异常。

表现
HTML 被截断、空文件、重复输出、模板残留、助手文本泄露、标签未闭合。
方向
加入静态完整性检查、HTML 解析检查和异常输出过滤。

评测链路本身存在噪声

分析

低分不一定全部来自模型能力，截图、裁剪和官方打分函数也会引入失败。

表现
点击前后截图尺寸不一致、交互裁剪为空、评测器中断后按 0 计入。
方向
建立失败归因层，把生成失败、交互失败、截图失败和评测失败分开统计。
布局先验先抽页面骨架和比例，再写 HTML/CSS。
状态建模显式描述动作前后状态和变化区域。
执行检查自动点击目标元素，确认页面确实变化。
渲染修复用截图差异驱动一到多轮代码修正。
失败归因把模型失败和评测链路失败拆开分析。
