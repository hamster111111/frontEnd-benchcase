任务、输入、输出和评测链路

WebGen-Bench 不是单页截图复现，而是从自然语言需求生成完整网站，再用 UI agent 和 VLM 评测网站功能与外观。

输入很短，测试很细

Benchmark 任务

输入
每个 app 给一段自然语言 instruction，描述网站类型、核心功能、用户操作和颜色约束。评测集还带有隐藏/保留的 ui_instruct 子任务，用于后续 UI evaluation。
输出
一个可运行的网站项目，通常包括生成 trace、zip、展开源码、依赖文件、启动脚本和运行截图。不是只交一个 HTML 片段。
UI 评测
WebVoyager 风格 UI agent 打开生成网站，逐条执行 647 个测试任务，输出 YES / PARTIAL / NO / START_FAILED。
外观评测
对每个 app 的页面截图做 1-5 分 appearance judge，关注成功渲染、内容相关性、布局和现代美观程度。

我们这次的关键差异

生成模型
论文主表没有 qwen3.5-9b；我们使用 OpenRouter 上的 qwen3.5-9b 作为 Bolt.diy 主体生成模型。
UI judge
论文 UI agent engine 使用 Qwen2.5-VL-32B-Instruct；本地最终合并中使用 qwen3-vl-plus 与 Mimo 修复 APIConnectionError 后的结果。
Appearance
论文用 GPT-4o 打外观分；我们使用 Mimo 对 100 张成功截图重评，000051 因依赖/启动问题没有截图。
本地兼容
生成路线对齐 Bolt.diy，但运行时有 Windows、Vite、静态页面和续跑兼容处理；这些会影响工程复现细节，不改变任务定义。

模型生成输入和评测任务不是一回事

以 case 000001 为例：生成阶段主要把 instruction 交给 Bolt.diy + 模型；ui_instruct 是官方保留的评测任务，后续由 UI agent 打开生成网站逐条执行。

Case 000001
生成阶段输入

这段需求会进入模型/agent，用来生成完整网站项目。

给模型
请实现一个用于生成股票报告的网站，提供股票信息和分析。网站需要支持搜索和汇总股票信息，并根据用户需求生成定制化股票报告。用户应能输入股票代码或名称，选择报告格式和内容，网站自动生成对应报告。报告应包含股票基础信息、市场趋势、财务数据等。背景色设为白色，组件颜色设为海军蓝。
需求类别数据管理 需求子类增删改查 / API 集成 / 数据可视化 应用类型分析平台 / 仪表盘
展开英文原文

Please implement a website for generating stock reports to provide stock information and analysis. The website should have the functionality to search and summarize stock information, and generate customized stock reports based on user requirements. Users should be able to input stock codes or names, select report formats and content, and the website will automatically generate the corresponding reports. The reports should include basic stock information, market trends, financial data, and more. Set the background color to white and the component color to navy.

Category: Data Management；Subcategories: CRUD Operations / API Integration / Data Visualization；Application Type: Analytics Platforms/Dashboards

ui_instruct 评测任务

这些任务不用于指导生成；评测时 UI agent 用它们检查网站功能、数据展示和设计约束。

评测用
01
输入有效股票代码，验证股票搜索功能。
系统应返回与该股票代码相关的信息和分析，包括基础信息、市场趋势和财务数据。
功能测试 搜索
英文原文

Verify the stock search functionality by entering a valid stock code.

The system returns relevant stock information and analysis corresponding to the entered stock code, displaying details such as basic stock information, market trends, and financial data.

02
选择不同报告格式和内容选项，测试报告定制功能。
系统应提供多种报告格式和内容定制选项，让用户可以按偏好选择。
功能测试 生成
英文原文

Test the report customization feature by selecting different report formats and content options.

The system provides options for various report formats and content customization, enabling users to choose according to their preferences.

03
输入股票名称而不是股票代码，搜索股票信息。
网站应能识别股票名称，并检索、展示相关股票信息和分析。
功能测试 搜索
英文原文

Input a stock name instead of a stock code to search for stock information.

The website successfully recognizes the stock name, retrieving and displaying the relevant stock information and analysis.

选择特定元素生成定制报告，例如市场趋势、财务数据。
生成的报告应准确包含用户选择的元素，并正确反映用户指定的内容。

Generate a customized stock report by choosing specific elements to include (e.g., market trends, financial data).

执行流程

从需求到分数的阶段拆解。

1读入 instruction

官方样本包含网站需求、类别和保留的 UI 测试任务。

2Bolt.diy 生成

模型在 agent scaffold 中生成完整项目和代码包。

3安装与启动

解压、安装依赖、启动本地服务，失败则进入 START_FAILED 或截图缺失。

4UI 子任务评测

UI agent 按 ui_instruct 执行操作并判断 YES/PARTIAL/NO。

5Appearance 评测

对页面截图进行 1-5 分外观判断，和 UI 分数分开分析。

指标说明

这些解释用于读 Table 3-5，和我们后面本地结果的列对齐。

Yes Rate

UI 子任务被评为 YES 的比例，表示网站完全满足该条测试要求。

Partial Rate

UI 子任务被评为 PARTIAL 的比例，表示部分满足要求，但功能、内容或交互仍有缺失。

No Rate

UI 子任务被评为 NO 的比例，表示没有达成该条测试要求，或页面没有可用入口。

Start Failed

网站启动失败、无法进入可测状态的比例。Table 5 的人工对齐表没有这一列。

Accuracy

论文主分数：(YES + 0.5 * PARTIAL) / Total × 100%。YES 权重为 1，PARTIAL 权重为 0.5。

Appearance Score

页面外观评分，范围 1-5，由 VLM 根据渲染成功、内容相关性、布局和现代美观程度给分。

Instruction Categories

网站生成需求所属类别：Content Presentation、User Interaction、Data Management。

Test Case Categories

评测子任务类别：Functional Testing、Data Display Testing、Design Validation Testing。

Testing Method

Table 5 中的 UI Agent 是自动浏览器评测，Manual 是人工复核结果。

Alignment Rate

Table 5 中 UI agent 结果与人工标注一致的比例，公式为 N_manual=agent / N_total × 100%。

结论：WebGen-Bench 有人持续打，但还没有被刷爆。

后续最强的直接结果大致在 50%-60% accuracy / FSR 区间；7B-8B 级模型经过 SFT、Step-GRPO、scaffold 或视觉反馈后可以到 40% 左右。它不是 nobody cares 的冷门 benchmark，但距离 90%+ 饱和还很远。

WebGen-Agent

arXiv 2025-09全量 101 / 647Accuracy + Appearance

核心做法是把网站生成改成多步闭环：代码生成、执行、截图反馈、GUI-agent 测试反馈、backtrack 和 select-best。它还用 step-level screenshot / GUI-agent score 做 Step-GRPO。

对我们很有参考价值：它证明“生成后视觉/交互反馈修复”能在 WebGen-Bench 上涨分，accuracy 31.8 -> 41.8，appearance 2.74 -> 3.96。

不能作为原榜单全量数字，但它非常支持我们的问题动机：Web app 生成难点在“从需求到可测试功能闭环”，TDD/浏览器反馈可以显著提高分数。

类别分析

这里先讲清楚 case、instruction category、test case category 的关系，再解释每个 category 具体在测什么。

User Interaction 最弱

Category 体系关系

WebGen-Bench 的类别不是同一个层级，读分数时要先分清分母。

Case / App 101 个网站需求

每个 case 是一个完整网站生成任务：给一段 instruction，要求模型产出一个可运行网站。

Instruction Category 给整个网站需求分类型

一个 case 通常只有一个主类别，用来描述这个网站需求本身属于内容展示、用户交互还是数据管理。

Test Case Category 给每条评测任务分类型

一个 case 会拆成多条 ui_instruct，每条评测任务再标成功能测试、数据展示测试或设计验证测试。

Instruction Category 是什么

它描述“这个网站需求本身属于哪种类型”，一个 case 通常对应一个主类。

Content Presentation 内容展示型
34.77%

核心是把信息组织成可读网页，例如机构介绍、课程/文章、产品目录、活动宣传、帮助文档等。它主要考察内容覆盖、页面结构、信息层级和基础视觉呈现。

28 个 case 174 条 ui_instruct 静态内容、导航、列表/详情、视觉排版
User Interaction 用户交互型
8.15%

核心是用户可以完成一串动作，例如注册登录、报名申请、购物下单、聊天咨询、提交表单、预约服务等。它最容易暴露状态流、表单校验和跨页面闭环问题。

49 个 case 313 条 ui_instruct 多步骤流程、按钮/表单、状态变化、错误处理
Data Management 数据管理型
15.63%

核心是围绕数据对象做搜索、筛选、增删改查、统计报表、图表或记录管理。它要求页面不只是有外壳，还要有稳定的数据模型和数据展示逻辑。

24 个 case 160 条 ui_instruct CRUD、搜索筛选、报表图表、列表/详情同步

Test Case Category 是什么

它描述“每条 ui_instruct 在检查什么能力”，同一个 case 下面会有多条不同类别的评测任务。

Functional Testing 功能测试
11.36%

逐条检查某个功能路径能不能跑通，例如搜索、生成、提交、登录、预约、导航、表单校验等。它是任务数最多的一类，也是 WebGen 里最低分的主战场。

339 条任务 52.40% task 占比 能不能完成用户动作
Data Display Testing 数据展示测试
19.89%

检查页面是否正确展示列表、详情、图表、历史记录、状态、结果页或推荐内容。它关注数据有没有出现、是否和任务目标一致、是否能被用户读到。

186 条任务 28.75% task 占比 有没有把数据正确展示出来
Design Validation Testing 设计约束测试
29.10%

检查 instruction 中明确给出的颜色、背景、组件视觉风格和布局要求是否被实现。它不一定要求复杂交互，但会惩罚空白页、样式不符和组件缺失。

122 条任务 18.86% task 占比 视觉约束是否按要求实现

读表结论

这里直接保留对方法设计有用的类别结论。

1

分数不能只按 case 看。一个网站需求会拆成多条 ui_instruct；一个空白页会让这个 case 下的多条功能、数据展示、设计验证任务一起失败。

2

Instruction Category 解释“网站是什么”。User Interaction 最低，说明带流程闭环的网站最难；Content Presentation 较高，说明静态信息组织相对容易。

3

Test Case Category 解释“评测在查什么”。Functional Testing 覆盖最多，也最低，说明真正卡分的不是单个页面长得像不像，而是功能路径能不能被 UI agent 跑通。

可优化方向

这里把 WebGen-Bench 带来的主要 insight 落成可做方法的方向，重点不是单纯 polish，而是从一句话需求到可运行网站的闭环。

方法动机候选

WebGen-Bench 的核心挑战：一句话产品需求，要生成一个完整可用的前端网站。

它和截图复现类 benchmark 不同。模型没有明确设计稿、组件树或交互前后状态，只拿到一段很短的 instruction，却要补全页面结构、角色、数据对象、用户流程、视觉约束和可运行工程。我们当前看到的失败，很多不是某个按钮写错，而是这个隐式产品规格没有被展开。

1需求展开

把一句话转成页面、角色、数据和验收点。

2规格检查

确认功能入口和测试任务能互相覆盖。

3代码生成

按结构化 spec 生成完整项目，而不是只写外壳。

4启动截图

先排除白屏、报错、占位页和资源缺失。

5浏览器验证

按任务逐条点击、输入、检查状态变化。

6定向修复

根据失败类型回写 spec 或代码并重跑。

Insight 1 一句话需求要生成完整前端系统。

WebGen 不给截图、组件树或交互状态，只给短 instruction。模型必须主动补全页面结构、角色、数据对象、流程和视觉约束。

可做方向：spec-first planner，把短需求展开成可检查的产品规格。

Insight 2 短 instruction 和细验收之间有鸿沟。

输入很短，但 ui_instruct 会按注册、搜索、提交、展示、颜色等细任务验收。模型如果不主动生成验收清单，就容易只写一个表层页面。

可做方向：把 instruction 展开成页面清单、组件清单、数据清单和测试点清单。

Insight 3 低分首先不是审美问题，而是可运行性问题。

大量失败来自白屏、只渲染背景色、运行/编译错误页或占位页。一个前置失败会让同一个 case 下多条 UI task 一起变成 NO。

可做方向：先做启动、截图、非空页面、控制台错误和资源完整性自检。

Insight 4 功能必须可发现，入口不存在就等于没有实现。

UI agent 通过浏览器使用页面，不读源码。注册、登录、发布、申请、消息、搜索等能力必须有清晰入口、可操作表单和反馈区域。

可做方向：把每条 ui_instruct 映射到导航项、按钮、输入框和结果区域。

Insight 5 User Interaction 的难点是状态闭环。

交互型网站低分不是因为按钮少，而是注册登录、申请、提交、消息、预约等流程没有保存状态、更新列表、显示反馈或区分角色。

可做方向：显式生成用户流程图或状态机，再用浏览器逐步执行验证。

Insight 6 Data Management 需要数据模型，不是 dashboard 外观。

很多页面像管理系统，但没有稳定的列表、详情、筛选、图表、记录更新和报表生成逻辑。数据类任务要求数据对象和页面状态真的存在。

可做方向：先生成 mock data schema、CRUD contract、数据流和展示视图。

Insight 7 UI 分和外观分会分叉。

有的页面功能路径能部分跑通，但视觉很差；也有页面只满足背景色却没有业务内容。功能正确和视觉质量需要分开诊断。

可做方向：browser task 结果修功能路径，VLM appearance 修视觉约束、布局、破图和现代感。

Insight 8 WebGen 适合做 agentic repair，而不是一次性生成。

失败信号很明确：启动失败、截图空白、找不到按钮、任务 NO、appearance 低分。它天然适合生成后执行检查，再按失败类型定向修复。

可做方向：生成-运行-截图-点击-诊断-修复的单 agent 或双 agent 闭环。

Insight 9 它和截图复现类 benchmark 互补。

Vision2Web / Interaction2Code 更像照着明确目标复现；WebGen 更像从需求构建产品。它能支撑“requirement-to-frontend-system”的研究叙事。

可做方向：证明框架不仅会复现界面，还能补全需求、构建系统并通过浏览器验证。
